### Top 5 NEW findings (not surfaced by prior audits)

1. **⚠️ `APP_URL` is documented in `.env.example` Section 3 but never consumed by any server module.** Either wire it into email-link generation / OAuth callback base, or drop the doc. The user's local file has `PUBLIC_URL` instead — also unused. **Outcome: no visible bug today, but a beginner setting `APP_URL` and expecting OAuth redirects to use it will be silently surprised.**
2. **⚠️ User's local `frontend/.env` uses `BICTORYS_BASE_URL`** but `frontend/src/lib/server/payments/provider-singleton.ts:57` reads `process.env.BICTORYS_API_URL`. Bictorys calls fall back to the default URL silently — that's why the audit brief said "user just did a final cleanup pass" and the kit "ships clean"; user-side env has a typo that the kit's docs don't catch.
3. **⚠️ User has `frontend/.env`, not `frontend/.env.local`.** The README/setup-kit instruct beginners to create `.env.local`; the Phase 0 audit probe `grep -q '^DATABASE_URL=postgresql://' frontend/.env.local` would report `UNSET` for this user even though the app boots fine.
4. **⚠️ `STATUS.md` L130-143 still describes Phase 6 as "in flight" + references `frontend/Dockerfile` (which no longer exists in the repo).** A beginner reading STATUS.md after README would be confused about Docker status.
5. **ℹ️ `CLAUDE.md` doc claims "559/559" tests** but actual is 555/555 (Sim 1 confirmed). Single-line doc drift. The audit brief acknowledged this — not a new finding strictly, but worth recording in the audit history.

### One-sentence ship-clean verdict

**The kit ships clean for v1.** Zero functional blockers, all simulations green (CI 4/4, build, dev boot + DB probe, git pristine), zero regressions vs. prior audits. The 5 ⚠️ items above are documentation drift (STATUS.md historical narrative, `APP_URL` orphan, user-side env naming typos) — none block "clone → plug DATABASE_URL → `pnpm dev` → ship".
